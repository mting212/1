## 1. 产品愿景

MeetFlow 致力于成为**最尊重双方时间的会议调度工具**。传统的调度工具往往只服务于组织者（"把空档全部暴露出去"），或只服务于受邀者（"随便点"），而 MeetFlow 在两者之间建立平衡：

- **组织者端**：精细控制可预约时段、会议密度和节奏，保护深度工作时间。
- **受邀者端**：直观的周视图界面，可叠加自己的日历，一眼找到互不冲突的时间，告别反复切换和来回确认。

> 设计哲学：**调度不是"填满日历"，而是"用最少的时间成本，让合适的人在对的时间相遇"。**

### 2.1 核心用户画像

|---|---|---|
| **独立顾问/自由职业者** | 客户咨询、项目讨论 | 时区混乱、来回邮件确认、多日历管理 |

- **MVP 阶段**：个人用户 + 小团队（1-20 人）

## 3. 核心功能

### 3.1 智能调度链接

> 这是产品的核心交互单元，类似 SavvyCal 的高保真周视图。

- **周视图日历**：受邀者看到以周为单位的时间网格，而非传统的"点日期→选时间"两步流程。
- **日历叠加（Calendar Overlay）**：受邀者可以连接自己的 Google / Outlook 日历，将自己的日程叠加在组织者的可用时段上，双方可用一目了然。
- **时区智能感知**：自动检测受邀者时区，所有时间以受邀者本地时间显示，避免跨时区混乱。

```
┌─────────────────────────────────────────────────┐
│  Meet with 张三                                   │
│                                                   │
│  ◄ May 25 - 31, 2026 ►        Your timezone: CST  │
│                                                   │
│  Mon 25    Tue 26    Wed 27    Thu 28    Fri 29   │
│  ┌────────┬────────┬────────┬────────┬────────┐  │
│  │ ████   │ ████   │ 08:00  │ 08:00  │ ████   │  │
│  │ ████   │ 09:00  │ 09:00  │ 09:00  │ ████   │  │
│  │ 10:00★ │ ████   │ 10:00★ │ ████   │ 10:00  │  │
│  │ 11:00  │ ████   │ 11:00  │ ████   │ 11:00  │  │
│  │ ████   │ ████   │ ████   │ 14:00  │ ████   │  │
│  │ 15:00  │ 15:00  │ ████   │ 15:00  │ 15:00  │  │

│                                                   │
│  ██ = busy    ★ = preferred    blank = available │
│  ── = your calendar overlay (from Google)         │
└─────────────────────────────────────────────────┘
```

### 3.2 可用性控制（可用性即策略）

| 控制项 | 说明 |
|---|---|
| **时间块（Time Blocks）** | 基于日历事件自动确定可用性；支持"反向时间块"——划定绝不可预约的时段 |
| **缓冲时间（Buffers）** | 会议前后自动锁定的间隙（如：会前 15 分钟、会后 10 分钟） |
| **每日/每周/每月上限** | 防止过度预约，可设置每日最多 4 场、每周最多 15 场等 |
| **提前期/滚动窗口** | 只能预约未来 N 天内的时段（如：未来 14 天），同时设置最短提前期（如：至少 2 小时后） |
| **会议时长限制** | 单次会议最短/最长时间（15 分钟、30 分钟、60 分钟等） |
| **首选时段（Preferred Slots）** | 将某些时段标记为"推荐"，引导受邀者优先选择，实现会议聚集而非散落 |

### 3.3 团队调度模式

#### 集体模式（Collective）

> 展示多个成员的可用时段**交集**，适合面试、项目启动会等需要全员参加的会议。

#### 轮询模式（Round Robin）

> 将会议请求轮流分配给团队成员，适合销售线索分配、客户支持分流。

#### 群组模式（Group）

> 允许多人同时预约同一个时段，适合网络研讨会、公开课、小组辅导。

### 3.4 会议投票（Meeting Polls）

> 不直接暴露可用性时使用。

- 组织者提出 N 个候选时间
- 受邀者投票选择自己方便的时间
- 组织者根据结果确定最终时间
- 支持"与/或"模式：必须全部出席时选交集；可以部分出席时选并集

### 3.5 自动化工作流

```
预约前:
  ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ 资格表单  │ → │ 付款确认  │ → │ 邮件通知  │
  └──────────┘    └──────────┘    └──────────┘

预约中:
  ┌──────────┐    ┌──────────┐
  │ 日历创建  │ → │ 视频链接  │
  └──────────┘    └──────────┘

预约后:
  ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ 提醒邮件  │ → │ 会后问卷  │ → │ CRM 同步 │
  └──────────┘    └──────────┘    └──────────┘
```

- **触发事件**：预约创建、预约修改、预约取消、预约前 N 小时
- **动作**：发送邮件/SMS、创建日历事件、生成视频会议链接（Zoom/Google Meet/Teams）、Webhook 回调、CRM 同步

### 3.6 品牌化与嵌入

- **自定义品牌**：Logo、颜色、Banner 图片、个性化欢迎语
- **自定义域名**：使用组织者自有域名托管预约页面（如 `meet.example.com`）
- **嵌入式组件**：以 iframe 或 Web Component 形式嵌入到自有网站
- **多语言支持**：预约页面支持中文、英文、日文等多语言界面

### 3.7 支付集成

- **Stripe 集成**：在预约时收取费用（咨询费、课程费）
- **定价配置**：自由定价 / 固定价格 / 按会议时长计费
- **退款与取消策略**：自定义取消截止时间和退款规则

### 3.8 AI 智能协调（差异化亮点）

> 这是超越 SavvyCal 等竞品的核心竞争力。

| AI 功能 | 描述 |
|---|---|
| **智能时间推荐** | 基于历史会议模式，自动推荐最佳的会议时段（避开你的"高专注时段"） |
| **自然语言调度** | "帮我在这周找个和张三、李四都方便的一小时" → 自动匹配并发送链接 |
| **会议摘要生成** | 预约后自动生成会议背景信息（参与者、时间、时区确认） |
| **冲突预测与建议** | 当常规偏好产生冲突时，AI 给出备选方案并说明权衡 |
| **智能分组** | 群组模式下，AI 根据受邀者时区自动分组，减少跨时区成员的痛苦时间 |

## 4. 用户体验设计

### 4.1 设计原则

1. **双向尊重**：组织者保护时间，受邀者获得便利，没有一方被牺牲。
2. **一步到位**：减少点击次数和页面跳转，关键操作在一个视图中完成。
3. **渐进式披露**：默认简洁，高级功能按需展开，不吓跑新手。
4. **跨时区友好**：时区信息无处不在且不易产生歧义。
5. **品牌可信赖**：白标化的预约页面让组织者看起来专业，而非"用了个第三方工具"。

### 4.2 用户旅程

#### 组织者首次使用

```
注册 → 连接日历 → 设置可用时段 → 创建第一个预约链接 → 分享 → 收到第一个预订
  │        │            │                │              │
  │    OAuth 授权   模板引导      一键复制链接    邮件/站内通知
  │   Google/Outlook  (± 缓冲时间)  或嵌入网站
  │   iCloud
```

#### 受邀者预约

```
打开链接 → 看到周视图 → (可选)叠加自己日历 → 点击合适的时段 → 填写信息 → 确认
  │            │              │                  │            │          │
  │    自动检测时区    一键 Google 登录      实时显示冲突  名称+邮箱   收确认邮件
  │    以本地时间显示   无需离开页面          ★ = 推荐时段  可选字段    自动创建日历事件
```

### 5.2 技术选型

| 层级 | 技术 | 选型理由 |
|---|---|---|
| **前端** | React + TypeScript + Tailwind CSS | 组件化、类型安全、快速 UI 开发 |
| **后端** | Go (核心服务) + Node.js (BFF 层) | Go 高性能处理调度逻辑，Node.js 灵活处理前端适配 |
| **数据库** | PostgreSQL 15+ | 强事务保证、JSON 支持、成熟的时间处理 |
| **缓存** | Redis | 预约时段缓存、会话管理、任务队列 |
| **消息队列** | Redis Streams / RabbitMQ | 工作流触发、异步通知 |
| **日历同步** | CalDAV / Google Calendar API / Microsoft Graph API | 多日历平台统一接入 |
| **视频会议** | Zoom API / Google Meet API / Teams API | 自动生成会议链接 |
| **支付** | Stripe Connect | 多币种、全球合规、平台化计费 |
| **基础设施** | Docker + Kubernetes | 容器化部署、自动扩缩容 |
| **监控** | Prometheus + Grafana + Sentry | 性能监控 + 错误追踪 |

### 5.3 关键架构决策

#### 日历同步策略

```
轮询 + Webhook 混合模式:
- Google Calendar: Webhook (Push Notification) 实时同步
- Outlook: Webhook 实时同步
- iCloud/CalDAV: 轮询同步（每 5 分钟）
- 所有变更写入事件队列，异步处理
```

#### 时区处理

```
统一存储: 所有时间以 UTC 存储
展示转换: 根据用户时区偏好实时转换
重复事件: 存储 iCalendar RRULE 格式
夏令时: 使用 IANA 时区数据库，自动处理 DST 转换
```

#### 预约冲突检测

```
"两阶段锁"机制:
1. 展示阶段: 读取缓存中的可用时段（可能有短暂不一致）
2. 提交阶段: 数据库事务 + 行级锁，二次验证无冲突后写入
3. 冲突时: 友好提示受邀者选择其他时段
```

## 6. 数据模型

### 6.1 核心实体关系

```
┌──────────────┐       ┌──────────────────┐
│     User     │1─────*│  CalendarAccount  │
│              │       └──────────────────┘
│  id          │
│  name        │       ┌──────────────────┐
│  email       │1─────*│  ScheduleLink    │
│  timezone    │       └────────┬─────────┘
│  avatar_url  │                │
└──────┬───────┘       ┌────────┴─────────┐
       │1              │                  │
       │               │*                 │*
┌──────┴───────┐  ┌────┴────────┐  ┌──────┴────────┐
│ Availability │  │ Booking     │  │ ScheduleRule  │
│   Rule       │  │             │  │ (缓冲、上限等) │
└──────────────┘  └──────┬──────┘  └───────────────┘
                          │
                          │*
                   ┌──────┴──────┐
                   │  Attendee   │
                   │  (受邀者)    │
                   └─────────────┘
```

### 6.2 关键表结构

```sql
-- 用户
CREATE TABLE users (
    id UUID PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,
    name VARCHAR(255) NOT NULL,
    timezone VARCHAR(64) DEFAULT 'Asia/Shanghai',
    avatar_url TEXT,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 日历账户
CREATE TABLE calendar_accounts (
    id UUID PRIMARY KEY,
    user_id UUID REFERENCES users(id),
    provider VARCHAR(32) NOT NULL,  -- 'google', 'outlook', 'icloud'
    access_token TEXT,
    refresh_token TEXT,
    calendar_id VARCHAR(255),
    sync_enabled BOOLEAN DEFAULT TRUE,
    last_synced_at TIMESTAMPTZ
);

-- 预约链接
CREATE TABLE schedule_links (
    id UUID PRIMARY KEY,
    user_id UUID REFERENCES users(id),
    slug VARCHAR(100) UNIQUE NOT NULL,     -- meetflow.com/zhangsan/30min
    name VARCHAR(255) NOT NULL,
    description TEXT,
    duration_minutes INT NOT NULL DEFAULT 30,
    custom_domain VARCHAR(255),
    branding JSONB DEFAULT '{}',
    is_active BOOLEAN DEFAULT TRUE,
    meeting_provider VARCHAR(32) DEFAULT 'zoom',
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 可用性规则
CREATE TABLE availability_rules (
    id UUID PRIMARY KEY,
    user_id UUID REFERENCES users(id),
    schedule_link_id UUID REFERENCES schedule_links(id),
    day_of_week SMALLINT NOT NULL,           -- 0=Sun, 6=Sat
    start_time TIME NOT NULL,
    end_time TIME NOT NULL
);

-- 预订记录
CREATE TABLE bookings (
    id UUID PRIMARY KEY,
    schedule_link_id UUID REFERENCES schedule_links(id),
    organizer_id UUID REFERENCES users(id),
    start_time TIMESTAMPTZ NOT NULL,
    end_time TIMESTAMPTZ NOT NULL,
    status VARCHAR(32) DEFAULT 'confirmed',  -- confirmed, cancelled, rescheduled
    attendee_name VARCHAR(255) NOT NULL,
    attendee_email VARCHAR(255) NOT NULL,
    attendee_timezone VARCHAR(64),
    notes TEXT,
    meeting_url TEXT,                         -- 自动生成的视频链接
    calendar_event_id VARCHAR(255),           -- 组织者日历中的事件 ID
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 缓冲设置
CREATE TABLE schedule_rules (
    id UUID PRIMARY KEY,
    schedule_link_id UUID REFERENCES schedule_links(id),
    rule_type VARCHAR(32) NOT NULL,          -- 'buffer_before', 'buffer_after',
                                                -- 'daily_limit', 'weekly_limit',
                                                -- 'min_notice_hours', 'max_future_days'
    rule_value INT NOT NULL
);
```

## 7. 功能路线图

### Phase 1：MVP（0-3 个月）

| 功能 | 优先级 | 说明 |
|---|---|---|
| 用户注册与日历连接 | P0 | Google / Outlook OAuth |
| 单一预约链接 | P0 | 基础周视图 + 时段选择 |
| 固定可用时段设置 | P0 | 每周重复规则 |
| 预约创建与确认 | P0 | 邮件通知 + 日历事件自动创建 |
| 时区自动检测 | P0 | 受邀者时区展示 |
| 自定义品牌色 | P1 | Logo + 颜色 |

**MVP 验收标准**：一个用户可以连接日历、创建预约链接、收到第一个预约。

| 功能 | 优先级 | 说明 |
|---|---|---|

| 首选时段 | P1 | 引导预约聚集 |
| 会议投票 | P1 | 群组时间协调 |
| 自定义域名 | P1 | 白标体验 |
| 嵌入组件 | P2 | iframe / Web Component |
| Stripe 支付集成 | P1 | 付费预约 |

### Phase 3：团队协作（6-9 个月）

| 功能 | 优先级 | 说明 |
|---|---|---|
| 轮询模式 | P0 | 销售团队分流 |
| 集体模式 | P0 | 多人可用交集 |
| 群组模式 | P1 | 一对多预约 |
| 团队仪表盘 | P1 | 团队整体视图 |

## 9. 竞争差异化

### 9.1 竞品格局

| 竞品 | 优势 | 劣势 |
|---|---|---|
| **Calendly** | 市场份额大、集成丰富、品牌认知高 | 界面传统、受邀者体验差（两步流程）、保护组织者不足 |
| **SavvyCal** | 日历叠加体验好、界面精美、尊重双方 | 无移动端 App、集成生态弱、无 AI 功能 |
| **Cal.com** | 开源可自托管、开发者友好 | 体验粗糙、非技术用户门槛高 |
| **TidyCal** | 一次性付费、价格低 | 功能有限、长期维护不确定 |
| **Google Calendar 预约** | 免费、原生集成 | 功能极简、不可定制 |

### 9.2 MeetFlow 的差异化定位

```
               尊重组织者
                   ▲
                   │    ★ MeetFlow
                   │   (AI + 双向尊重)
                   │
    Calendly ──────┼────── SavvyCal
    (组织者友好)    │      (受邀者友好)
                   │
                   │  Cal.com
                   │  (开发者友好)
                   │
                   └──────────────────► 尊重受邀者
```

| 差异化维度 | MeetFlow 优势 |
|---|---|
| **AI 驱动** | 智能时间推荐、自然语言调度、冲突预测——竞品无此能力 |
| **双向尊重** | 不偏向组织者也不偏向受邀者，在两端找到最佳平衡 |
| **团队模式** | 轮询、集体、群组三种模式覆盖所有团队场景 |
| **中国本土化** | 支持微信/钉钉/飞书日历，符合国内用户习惯 |
| **开放 API** | 从 Day 1 就设计完善的 API，支持生态扩展 |
| **移动优先** | 原生考虑移动端体验，而非简单响应式适配 |

### 9.3 核心竞争壁垒

1. **AI 调度算法**：随着数据积累，推荐越来越精准，形成数据飞轮
2. **日历同步基础设施**：跨平台日历同步的技术复杂度是天然门槛
3. **品牌化生态**：自定义域名 + 嵌入组件 = 组织者品牌资产沉淀在平台
4. **网络效应**：受邀者→组织者的转化路径

### B. 设计决策记录

| 决策 | 理由 |
|---|---|
| 周视图而非月视图 | 移动端友好、信息密度适中、符合"近期预约"的使用习惯 |
| UTC 存储 + 本地展示 | 避免夏令时和跨时区 bug |
| PostgreSQL 而非 NoSQL | 调度系统需要强事务保证（防双订） |
| Go 后端 | 高并发场景下性能可靠，适合调度计算密集型任务 |
| 自建日历同步而非依赖第三方 | 降低外部依赖风险，保证数据主权 |

### C. 风险与缓解

| 风险 | 影响 | 缓解策略 |
|---|---|---|
| Google/Outlook API 变更 | 日历同步中断 | 多协议兼容（CalDAV）+ 变更监控 + 快速适配 |
| 预约冲突（双订） | 体验灾难 | 两个阶段锁 + 数据库行锁 + 补偿机制 |
| 竞品（Calendly）碾压 | 用户获取困难 | 聚焦差异化（AI、双向尊重、本土化）而非正面竞争 |
| 用户隐私顾虑 | 信任危机 | 最小化数据收集、透明隐私政策、SOC 2 认证 |
| AI 功能不达预期 | 用户失望 | 渐进式交付，先做确定性规则再做概率性推荐 |
